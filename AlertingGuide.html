Alerting Setup in NinjaOne Platform

1. API Integration for Tickets

The API integration for tickets in NinjaOne is configured as a one-way integration. This means that:

  • Closing a ticket in ConnectWisePSA does not clear the condition in NinjaOne.
  • Clearing the condition in NinjaOne will automatically close the corresponding ticket in ConnectWisePSA.

2. Alert Changes

Alert changes that are not specific to a single device override must be made at the Parent policy level. This ensures consistency and centralized management of alert configurations.

2.1 Parent Policy Level Changes

  • All global alert settings and modifications should be applied at the Parent policy level.
  • This includes changes to alert thresholds, notification settings, and response actions.

3. Alert Criteria

Alerts in NinjaOne are based on various criteria, including hardware metrics, time-based conditions, Windows Events, script output evaluations, software installations, and custom fields.

3.1 Hardware Metrics and Time

  • Example: CPU usage at 90% for 15 minutes.
  • Configure alerts to trigger based on sustained hardware performance metrics.

3.2 Windows Events

  • Monitor specific Windows Event IDs and trigger alerts based on their occurrence.
  • Customize event-based alerts to match critical system events.

3.3 Script Output Evaluation

  • Run scripts to evaluate system conditions and trigger alerts based on the output.
  • Example: A script that checks disk space and triggers an alert if space is below a certain threshold.

3.4 Software Installed

  • Set up alerts for the installation or uninstallation of specific software.
  • Monitor changes in software inventory and trigger alerts accordingly.

3.5 Custom Fields

  • Use custom fields to define specific conditions for alerts.
  • Example: Monitoring custom attributes like warranty expiration dates and triggering alerts based on those fields.
