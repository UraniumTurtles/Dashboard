Monitoring System Setup

1. Parent Policies

Parent policies encompass all conditions and scripts/automations necessary for the monitoring system. These policies ensure that the system operates efficiently and effectively.

1.1 Conditions

  • Define the criteria for monitoring various network devices and systems.
  • Specify thresholds for alerts and notifications.
  • Establish rules for normal and abnormal behavior.

1.2 Scripts/Automations

  • Automate routine monitoring tasks to reduce manual intervention.
  • Implement scripts for data collection, analysis, and reporting.
  • Ensure scripts are regularly updated and maintained.

2. Child Policies

Child policies are specialized policies based on specific needs such as patching and AV/EDR (Antivirus/Endpoint Detection and Response).

2.1 Patching Policies

  • Schedule regular patching windows.
  • Define critical and non-critical patches.
  • Automate patch deployment and verification.

2.2 AV/EDR Policies

  • Set up real-time monitoring for malware and threats.
  • Define response actions for detected threats.
  • Regularly update AV/EDR definitions and rules.

3. Network Monitoring System (NMS)

The NMS is set up on a central server to monitor all network devices. This centralization ensures comprehensive visibility and control.

3.1 Central Server Configuration

  • Install and configure NMS software on the central server.
  • Ensure the server has adequate resources (CPU, memory, storage).
  • Set up secure access controls and user permissions.

3.2 Device Monitoring

  • Add all network devices to the NMS.
  • Configure device-specific monitoring parameters.
  • Set up dashboards and reports for real-time monitoring.

4. Overrides

Any overrides must have a note on the device explaining the reason for the override, along with a supporting ticket number.

4.1 Documentation of Overrides

  • Create a standardized format for override notes.
  • Ensure each note includes the reason for the override and the ticket number.
  • Regularly review and audit overrides for compliance.

5. After-Hours Alerts

After-hours alerts are managed based on severity and client requirements. Alerts are sent to a corporate email, which is then forwarded to an answering service.

5.1 Severity-Based Alerts

  • Define alert severity levels (e.g., critical, high, medium, low).
  • Configure alert thresholds for each severity level.
  • Ensure critical alerts are prioritized for immediate action.

5.2 Client-Based Alerts

  • Customize alert settings based on client requirements.
  • Ensure client-specific alerts are routed appropriately.
  • Maintain a contact list for after-hours support.

5.3 Email Forwarding

  • Set up a corporate email for receiving alerts.
  • Configure email forwarding to the answering service.
  • Ensure the answering service is equipped to handle alerts promptly.
